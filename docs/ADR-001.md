# ADR-001 — Identity and Invitation Model without Email Requirement (STRICT + NIC Anchor)

**Status:** Approved  
**Date:** 2026-03-27  
**System/Repo:** `igrp_platform_access_management`  

## 1. Context

**AUTENTIKA** is the Identity Provider (IdP) based on **WSO2 Identity Server 5.11**, supporting **OpenID Connect (OIDC)** and issuing **JWT tokens** (ID Token / Access Token).  
The tokens and/or the `/userinfo` endpoint provide relevant identification *claims*, such as: `sub`, `email`, `phone_number`, `NIC` and `auth_method`.  

The goal is to eliminate the **email** requirement and allow a user to login/authenticate via **Mobile Phone Number** or **CNI (NIC)** as well, keeping everything linked to the **same user**.

In parallel, the target architecture for the service must follow the **No-Adapter** principle: the service must **not** perform synchronizations or administrative operations in the IAM; the only interaction with the IAM is **JWT validation + claim extraction** at authentication time.  

## 2. Decision

### 2.1. Canonical Identity (Anchor)

- The canonical identifier of the user within the service domain is the **NIC**, obtained from the `sub` claim (in AUTENTIKA, `sub` is always the NIC).  
- **Email** and **Phone** become **secondary** (optional) identifiers, and are never mandatory metrics to exist or authenticate in the service domain.  

### 2.2. Invite-only (Option A)

- **No** user is automatically created in the DB based solely on authentication (JIT provisioning is disabled).  
- If a user successfully authenticates at the IdP (valid JWT), but **does not exist** in the DB (`users.nic` is missing), the service **blocks** access and requires a **pending invitation + acceptance**.

### 2.3. Invitation Acceptance in STRICT mode

- Invitation acceptance must be validated **exclusively** by the primary identifier associated with the **authentication method** used for acceptance (`auth_method` claim).  
- The invitation must explicitly (or implicitly) define the allowed method and the type of identifier to be used for matching.

#### 2.3.1. Official Mapping (AUTH_METHOD → Primary Identifier)

- `CMD` → `phone_number` (Invitations via **PHONE**)  
- `CNI` → `sub` (= NIC) (Invitations via **NIC**)  
- Custom credentials → `email` (Invitations via **EMAIL**)  

### 2.4. No-Adapter (No IAM Admin)

- The service does **not** call IAM admin APIs to resolve/create users, roles, departments, or synchronize state.
- Identity correlation and invitation acceptance rely entirely on **claims** present in the JWT (and/or `/userinfo`).

## 3. Consequences

### 3.1. Benefits

- Email is no longer a requirement for login/authentication in the service domain.
- Identity is anchored to a strong and stable identifier (**NIC** via `sub`).
- Reduced coupling to the IAM and greater resilience (aligned with No-Adapter).
- Auditability and predictability: invitations are accepted deterministically (STRICT).

### 3.2. Costs / Trade-offs

- The user must accept the invitation using the correct method (e.g., an email invitation requires credentials; a phone invitation requires CMD).
- There may be initial friction in transitioning from “email-centric” systems.

## 4. Considered Alternatives (Rejected)

### 4.1. FLEX (Accept via any available claim)

Rejected as it increases ambiguity and the risk of improperly linking identifiers.

### 4.2. JIT Provisioning (Automatically create user on first login)

Rejected to maintain strict access control (invite-only).

## 5. Operational Rules

### 5.1. Gatekeeper (Invite-only)

- If `users.nic` does not exist for the JWT's `sub`, respond with a standardized error:

```json
{
  "error": "INVITE_REQUIRED",
  "message": "Unregistered user. You must accept an invitation to enable access."
}
```

### 5.2. Invitation Acceptance (STRICT)

- Validate JWT.
- Extract `sub` (NIC) and `auth_method`.
- Determine the primary identifier according to the mapping.
- Validate that:
  - The invitation is `PENDING`
  - The `auth_method` is allowed for the invitation
  - The invitation's `identifier_type` matches the primary identifier type
  - The invitation's `identifier_value_normalized` equals the normalized primary claim value
- Create user by NIC (if not exists) and assign roles in the DB.
