## 1. Current Situation

* **Menus (`t_menu_entry`)** are currently associated with **permissions** via the table `t_menu_entry_permission`.
* **Resource items (`t_resource_item`)** are directly linked to a single `permission_id`.

This design mixes role-based and permission-based associations inconsistently:

* Menus require permissions (indirect role access).
* Resource items allow only a single permission reference.

---

## 2. Proposed Changes

### 2.1 Menus: Move from Permission Association → Role/User Association (RBAC)

* **Remove `t_menu_entry_permission` and `t_menu_entry_permission_aud`.**
* **Introduce new associations:**

    * `t_menu_entry_role (menu_entry_id, role_id)`
    * `t_menu_entry_user (menu_entry_id, user_id)` (optional, for direct user overrides)

This enforces that **menus are controlled by roles (or users)**, not by fine-grained permissions.
Roles will still hold permissions (via `t_role_permission`), but **menus remain role-based only**, simplifying access checks at UI level.

#### New Tables:

```sql
create table t_menu_entry_role (
    menu_entry_id integer not null
        references t_menu_entry,
    role_id integer not null
        references t_role,
    primary key (menu_entry_id, role_id)
);

create table t_menu_entry_user (
    menu_entry_id integer not null
        references t_menu_entry,
    user_id integer not null
        references t_user,
    primary key (menu_entry_id, user_id)
);
```

---

### 2.2 Resources: RBAC + ABAC

Resource access requires **role-based** and **attribute-based** control.

#### Current:

* `t_resource_item.permission_id` links one item to a single permission.

#### Proposed:

1. **Support RBAC**: Introduce `t_resource_item_role` (role → resource item).
2. **Support ABAC**: Keep `t_resource_item.permission_id`, but extend with attribute conditions (e.g., department, context, or JSON-based rules).

#### New Tables:

```sql
-- Roles associated with resource items (RBAC)
create table t_resource_item_role (
    resource_item_id integer not null
        references t_resource_item,
    role_id integer not null
        references t_role,
    primary key (resource_item_id, role_id)
);

-- Extend ABAC by separating conditions from static permission_id
create table t_resource_item_abac (
    id integer generated by default as identity primary key,
    resource_item_id integer not null
        references t_resource_item,
    attribute_key varchar(255) not null,
    attribute_value varchar(255) not null
);
```

---

## 3. Justification & Decision Rationale

### 3.1 Why change menu association from permissions to roles?

* **Simplicity:** Menus are **UI-level access points**, best controlled by roles (RBAC).
* **Consistency:** Users get menus based on their assigned roles, not low-level permissions.
* **Performance:** Checking role membership is faster and clearer than evaluating many permissions.
* **Alignment with best practices:** Roles represent “who can see what menu,” permissions represent “what actions are allowed inside.”

### 3.2 Why add RBAC + ABAC for resources?

* **RBAC** ensures standard role-based access.
* **ABAC** allows finer control (e.g., “User views a component within the page only if permission = X”).
* This combination gives flexibility: roles grant baseline access, attributes refine conditions.

## App Menus Fetching Changes

### Current Logic

* Collects **permission IDs** from user roles.
* Filters menus by intersecting with `menu.getPermissions()`.

### New Logic (After RBAC change for menus)

* Collect **roles** assigned to the user.
* Check if a `MenuEntryEntity` contains any of these roles via `menu.getRoles()`.
* **No more filtering by permissions for menus**.

---

### Updated `GetAppMenusQueryHandler`

```java
@IgrpQueryHandler
public ResponseEntity<List<MenuEntryDTO>> handle(GetAppMenusQuery query) {
    String username = authenticationHelper.getPreferredUsername();
    LOGGER.info("Fetching current user with username: {}", username);

    Optional<IGRPUserEntity> optionalUser = igrpUserRepository.findByUsername(username);
    if (optionalUser.isEmpty()) {
        LOGGER.warn("No user found with username: {}", username);
        return ResponseEntity.status(HttpStatus.NOT_FOUND).build();
    }

    IGRPUserEntity user = optionalUser.get();

    // Step 1: Get application by appCode
    Optional<ApplicationEntity> optionalApp = applicationRepository.findByCodeAndStatusNot(query.getAppCode(), Status.DELETED);
    if (optionalApp.isEmpty()) {
        LOGGER.warn("No application found with code: {}", query.getAppCode());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).build();
    }

    ApplicationEntity app = optionalApp.get();

    // Step 2: Collect user's active roles
    Set<RoleEntity> userRoles = user.getRoles().stream()
            .filter(role -> role != null && role.getStatus() == Status.ACTIVE)
            .collect(Collectors.toSet());

    if (userRoles.isEmpty()) {
        LOGGER.warn("User {} has no active roles assigned", username);
        return ResponseEntity.ok(Collections.emptyList());
    }

    // Step 3: Fetch all active menus for the app
    List<MenuEntryEntity> allMenus = menuEntryRepository.findByApplicationIdAndStatus(app, Status.ACTIVE);

    // Step 4: Filter menus based on role association
    List<MenuEntryDTO> accessibleMenus = allMenus.stream()
            .filter(menu -> {
                Set<RoleEntity> menuRoles = menu.getRoles();
                if (menuRoles == null || menuRoles.isEmpty()) {
                    return false; // no role restriction defined → deny by default
                }
                return menuRoles.stream()
                        .filter(role -> role.getStatus() == Status.ACTIVE)
                        .anyMatch(userRoles::contains);
            })
            .map(menuEntryMapper::toDTO)
            .collect(Collectors.toList());

    LOGGER.info("User {} has access to {} menus in application {}", 
                username, accessibleMenus.size(), query.getAppCode());

    return ResponseEntity.ok(accessibleMenus);
}
```

---

### Key Updates Made

1. **Removed permission filtering logic**.
2. **Replaced `menu.getPermissions()` with `menu.getRoles()`**.
3. **User’s effective access is now role-based for menus**.

## Resource Access Check

## Current Behaviour

* Each `ResourceItemEntity` has a `permissionId`.
* Access control relies on that permission alone (no role support, no ABAC invocation).

---

## New Behavior

1. **RBAC (roles in JWT):**

    * Use Spring Security’s **`@PreAuthorize("hasRole('...')")`** or equivalent.
    * Roles are resolved from the JWT, no DB lookup needed.
    * Mapping: `ResourceItemEntity.roles → JWT roles` via IAM adapters synchronization with provider.
    
2. **ABAC (attribute-based via external service):**

    * If a `permissionId` is configured for the `ResourceItemEntity`, resolve its `name` (e.g.: maintain_vehicle).
    * Call the **Access Management Client**:

      ```java
      @PutMapping("/vehicle/{id}")
      @PreAuthorize("@igrpAuthorization.checkPermission(authentication, 'vehicle:' + #id, 'maintain_vehicle')")
      public ResponseEntity<?> updateVehicle(@PathVariable String id, @RequestBody VehicleDto dto) {
          // update logic
      }
      ```
    * If the client confirms, access is granted.
    * Only **one permission** is supported per resource item (`permissionId` remains).

#### RBAC Check (default Spring Security)

In controllers/services, you can now use:

```java
@PreAuthorize("hasRole('ADMIN') or hasRole('MANAGER')")
public ResponseEntity<?> secureEndpoint() {
    // business logic
}
```

The JWT’s `roles` claim is automatically mapped to Spring Security’s `GrantedAuthorities`.

### Summary of Changes Needed

1. **Keep `permissionId`** for ABAC.
2. **Add RBAC support** by checking user roles (via JWT & Spring Security `@PreAuthorize`).
3. **ABAC integration**: introduce a service that calls the Access Management client with the `permissionName`.
4. **One permission per resource item** rule stays in place (no collection of permissions).
